# /feature

**模板分类**: Action

使用 `$ARGUMENTS` 中传递的问题上下文起草功能实施计划。

## 指令

- **首先验证问题标签**: 运行 `gh issue view <issue-number> --json labels` 确保问题有所有类别的标签
- 在 `docs/specs/` 下创建新的 markdown 计划，命名为 `feature-<issue-number>-<slug>.md`
- 从问题标题使用 3-6 个小写、连字符分隔的单词构建 `<slug>`
- 在计划顶部引用 `$ARGUMENTS` 中的问题元数据用于追溯
- 遵循仓库 git 流程
- 按以下确切格式填充

## 计划格式

```md
# 功能计划: <简洁名称>

## 概述
- 问题
- 期望结果
- 非目标

## 技术方法
- 架构说明
- 要触及的关键模块
- 数据/API 影响

## 相关文件
- <path> — <为什么此文件重要>
### 新文件
- <path> — <目的>

## 任务分解
### 阶段 1
- <基础任务>
### 阶段 2
- <实施任务>
### 阶段 3
- <集成 & 清理>

## 逐步任务
### <有序任务组>
- <按执行顺序的可操作项>
- 以重新验证和推送的任务组结束

## 风险与缓解
- <风险> → <缓解>

## 验证策略
- 自动化测试
- 手动检查
- 发布护栏

## 验证命令
- lint
- typecheck
- test --filter integration
- test
- build
```

## 提交消息验证

此功能的所有提交都将被验证。确保提交消息：
- 遵循 Conventional Commits 格式
- **避免元评论模式**
- 使用直接陈述

## 报告

**仅**返回计划文件路径作为纯文本单行。
