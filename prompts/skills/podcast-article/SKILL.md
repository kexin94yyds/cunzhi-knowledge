# 播客和公众号文章工作流

将语音录入内容转化为播客稿和公众号文章的完整工作流。

**核心原则：分步进行，每一步都要用户确认。不要一次性生成！**

---

## 工作流概览

```
语音转文字原稿 → 结构化原文稿(确认) → 播客稿(确认) → 公众号文章
```

---

## 第一步：原稿 → 结构化原文稿

用户提供语音转文字的原稿后，AI 先输出**结构化原文稿**：

- 提取核心观点，分章节整理
- 使用层级结构（一、二、三... + 小标题 + 要点）
- 保留原稿的核心内容

**范例**：参见 [example-structured.md](references/example-structured.md)

**等待用户确认结构是否 OK。**

---

## 第二步：结构化原文稿 → 播客稿

结构确认后，AI 将其**扩写成播客稿**：

- 口语化，适合讲述
- 有节奏感，短句为主
- 有互动感，可加问题收尾
- 分章节，每章有小标题

**风格特点**：
- 开场引入：用一个画面/场景切入
- 中间展开：每个观点用故事或例子支撑
- 结尾收束：用问题或金句收尾

**范例**：参见 [example-podcast.md](references/example-podcast.md)

**等待用户确认播客稿是否 OK。**

---

## 第三步：播客稿 + 结构化原文稿 → 公众号文章

播客稿确认后，AI **结合播客稿和结构化原文稿**写成公众号文章：

**提示词**：
```
结合播客稿和结构化原文稿，写成一篇公众号文章
```

**公众号文章特点**：
- 适合阅读的文章格式
- 段落分明，金句突出（加粗）
- 小标题清晰，分割线分隔章节
- 比播客稿更完整、更深入

**范例**：参见 [example-article.md](references/example-article.md)

---

## 注意事项

1. **不要一次性生成**：每一步都要等用户确认
2. **保持原稿风格**：不要过度改写用户的表达
3. **结构先行**：先确认结构，再写内容
4. **公众号 = 播客稿 + 结构化原文稿**：融合两者的优点
5. **模仿范例风格**：但不要完全拘泥，根据内容灵活调整
