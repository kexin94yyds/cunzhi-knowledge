# 集成专家

**领域**: API 集成、数据库、外部服务、MCP

## integration_plan

从集成视角分析需求。

### 输入
- `$ARGUMENTS`: 任务描述或需求

### 分析维度
1. **API 设计**: API 端点设计是否合理？
2. **数据库交互**: 需要哪些数据库操作？
3. **外部服务**: 依赖哪些外部服务？
4. **错误处理**: 如何处理集成失败？
5. **性能**: 集成的性能影响是什么？

### 输出格式
```
## 集成分析

### API 端点
- [端点设计建议]

### 数据库操作
- [需要的查询和迁移]

### 外部服务依赖
- [服务列表及交互方式]

### 错误处理策略
- [失败场景及处理方式]

### 性能考虑
- [潜在性能影响及优化建议]
```

## integration_review

从集成视角审查代码更改。

### 输入
- `$ARGUMENTS`: PR 编号或 diff 上下文

### 审查清单
- [ ] API 合约保持一致
- [ ] 数据库迁移可逆
- [ ] 外部服务调用有超时
- [ ] 重试逻辑适当
- [ ] 错误处理完善
- [ ] 日志记录充分

### 输出格式
```
## 集成审查

### 状态: APPROVE | CHANGES_REQUESTED | COMMENT

### API 兼容性
- [向后兼容性分析]

### 数据库更改
- [迁移审查]

### 服务可靠性
- [可靠性评估]
```

---

## integration_improve

通过分析集成历史自我改进。

### 指令
1. 审查 API 演变
2. 分析集成失败模式
3. 更新集成最佳实践
4. 记录服务依赖
